Overview

RheinEnergie requires an enterprise agentic-AI platform covering the full lifecycle of agent development, orchestration, deployment, and operations — integrated with their existing MuleSoft gateway, observed via Datadog, and governed under DSGVO/GDPR compliance.

Microsoft's answer spans three pillars:
  • Microsoft Foundry — Build, orchestrate, deploy & operate agents (11,000+ models, Agent Service, tracing, evaluations)
  • Azure API Management AI Gateway — Central control layer for models, agents & tools (unified model API, MCP/A2A, token governance)
  • Azure Governance & Compliance — Entra ID, Purview, Content Safety, EU Data Boundary, FinOps toolkit

AI Lifecycle Plattform

A single model catalog of 11,000+ models underpins every stage.

AI Gateway & Integration
Central control layer integrated with the existing architecture (esp. MuleSoft API Gateway).
The Azure API Management AI gateway is the central control layer for model, agent and tool traffic. Its unified model API exposes multiple backends (OpenAI, Anthropic, Google Vertex, Amazon Bedrock, Foundry) through one OpenAI-compatible endpoint, with automatic format translation and governance policies applied once across all models. It natively manages remote MCP servers and A2A agent APIs. It complements the existing MuleSoft gateway: MuleSoft's AI connectors (MCP, A2A, LLM) expose RheinEnergie's enterprise systems as governed tools to agents, while APIM governs the model/agent layer.

3
Observability
End-to-end logging, tracing and monitoring over Datadog (OpenTelemetry-based).
Foundry emits telemetry using OpenTelemetry semantic conventions and stores traces in Azure Monitor Application Insights; server-side tracing turns on automatically with no code changes and captures latency, exceptions, token counts, tool calls and (optionally) prompt content. Because the pipeline is OpenTelemetry-based, the same traces export to Datadog. Datadog additionally ships native integrations for Azure OpenAI and Azure AI Foundry plus LLM Observability — end-to-end latency, errors, I/O messages and token usage.

4
FinOps
Granular cost measurement & control — per use case (across agents) and per agent.
Token usage is metered natively — Foundry / Azure OpenAI emit input, output and total token counts per deployment. The APIM AI gateway's emit-token-metric policy attaches custom dimensions (use-case ID, agent ID, team) to every call, so cost is attributable per use case (across multiple agents) and per individual agent. The token-limit policy enforces budgets/quotas, and the Microsoft FinOps toolkit / FinOps hubs (FOCUS) consolidate and analyze the spend.

Governance
Central policy control incl. security, access, limits and compliance.
Central policy control spans identity, access, limits, safety and compliance: APIM applies security and governance policies once across every model; Azure Policy controls which model publishers may be deployed; Microsoft Entra ID (including agent identity / Entra Agent ID) governs authentication and access; Microsoft Purview provides data governance and compliance; Azure AI Content Safety adds guardrails. Agent Tracking
Transparent traceability of agent-to-agent (A2A) interactions.
Foundry's OpenTelemetry tracing records each agent run — tool calls, run steps, inputs/outputs and hand-offs — giving transparent traceability of agent-to-agent flows. At the gateway, APIM natively manages A2A agent APIs and MCP servers, so cross-agent interactions are mediated, logged and governed centrally, building on the open MCP and A2A standards (also supported by MuleSoft's connectors).

7
Data Privacy / DSGVO Layer
Abstraction layer separating metadata from content — analyze communication without accessing prompt contents.
By default, Foundry tracing captures only metadata (model name, token counts, latency, status) and does not record message content. Content recording is strictly opt-in via explicit flags (both default false). You can therefore analyze communication flows, volumes, latency and cost without accessing prompt contents — exactly the requested abstraction. This is reinforced by the EU Data Boundary (customer & pseudonymized personal data stored/processed in the EU/EFTA; GDPR Art. 4(5) pseudonymization of system logs), Foundry data zones, and Microsoft Purview. Microsoft does not use customer data to train shared models.

Multi-Provider Architektur
Support for multiple LLM providers — Microsoft Foundry, third-party, local models, own platforms.
Foundry's model catalog spans OpenAI, Anthropic (Claude, now GA on Azure), Meta Llama, Mistral, DeepSeek, xAI Grok, Cohere, Hugging Face, NVIDIA and more — all under one Azure identity, billing and governance. Model router exposes many providers through a single deployment; Foundry Local runs open models on-device / on-prem for private, low-latency use; and APIM's unified model API brings third-party and self-hosted ("own platform") endpoints under the same OpenAI-compatible interface.

Cost Efficiency
Technical mechanisms for optimization — routing, limiting, efficient model selection.
Model router analyzes each prompt in real time and routes to the most cost-effective model that still meets quality, with Balanced / Cost / Quality modes and reasoning-vs-non-reasoning selection — one deployment, lower spend. At the gateway, token-limit policies cap consumption, semantic caching avoids repeat inference, and load balancing spreads traffic across backends. Prompt caching and PTU-vs-pay-as-you-go options further optimize unit cost.

FinOps — Cost Attribution & Control

How cost attribution works end-to-end:
  • Per-agent metering: emit-token-metric policy tags every call with agent-id dimension → filter in Azure Monitor
  • Per-use-case metering: Same policy tags calls with use-case-id → aggregate across multiple agents
  • Budget enforcement: token-limit policy caps TPM per subscription/team/agent
  • Cost optimization: Model router selects cheapest model meeting quality bar; semantic caching avoids repeat calls
  • Reporting: FinOps toolkit / FinOps hubs consolidate Azure AI spend in FOCUS format for executive dashboards

Observability — OpenTelemetry + Datadog

Telemetry flow:
  • Foundry Agent Service emits OpenTelemetry traces automatically (server-side, zero-code)
  • Traces include: latency, token counts, tool calls, run steps, exceptions, hand-offs
  • Content recording is opt-in (default: metadata only → DSGVO compliant)
  • Traces stored in Azure Monitor Application Insights
  • Same OTel pipeline exports to Datadog via standard OTLP exporter
  • Datadog adds native Azure OpenAI + Foundry integrations + LLM Observability dashboard
